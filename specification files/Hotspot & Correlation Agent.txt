1. Hotspot & Correlation Agent – Role in the System
1.1 Purpose
The Hotspot & Correlation Agent takes multi‑modal signals for a case:

TOD Agent’s posterior distribution over time of death.

Anomaly windows from the Timeline & Anomaly Agent.

CCTV tracks, phone events, IoT events around those windows.

Key autopsy/TOD findings (e.g., estimated TOD, injury types, manner-of-death probabilities).

and fuses them into a small set of “hotspots”: intervals where “something important” likely happened, with a score and a transparent explanation.

Conceptually, this is a multi-modal fusion & hotspot identification layer analogous to:

Integrated multi‑model data fusion approaches in forensic / security systems.

Event correlation engines in incident detection platforms that group alerts into incidents.

Statistical/Bayesian hotspot detection frameworks that identify regions/time windows with multiple signals pointing to the same event.

The output drives:

The Hotspots view in the UI.

Inputs to the Causal Graph/Hypothesis Manager (nodes & edges).

Reasoning Theater / Replay sequences.

2. Data Contracts
2.1 Inputs
Per case, the agent consumes:

ts
HotspotFusionInput {
  caseId: string;

  todResult: TodResult;               // from TOD Agent
  anomalyWindows: AnomalyWindow[];    // from Timeline & Anomaly Agent
  timelineBuckets: TimelineBucket[];  // for context (main timeline)
  keyEvents: KeyEvent[];              // curated events (CCTV, phone, IoT, scene)
  hypotheses: HypothesisSummary[];    // NATURAL / ACCIDENT / SUICIDE / HOMICIDE distribution
}
We already defined TodResult, AnomalyWindow, TimelineBucket, and HypothesisSummary earlier.

2.1.1 KeyEvent
Key events are “coarse, important” events extracted by upstream agents (CCTV, phone, etc.):

ts
KeyEvent {
  id: string;
  caseId: string;
  timestamp: string;
  type: "CCTV_PERSON_SEEN" | "PHONE_CALL" | "APP_SESSION" | "IOT_DOOR" | "SCENE_EVENT" | "MANUAL_MARKER";
  severity: "INFO" | "INTERESTING" | "CRITICAL";
  description: string;
  attributes: Record<string, any>;   // e.g., cameraId, contactId, location, duration, etc.
}
Some of these might be elevated outputs from other agents (e.g. CCTV agent flags “person X enters building alone”).

4. Candidate Interval Generation
The agent must create candidate hotspot windows before scoring them.

4.1 Sources for Candidates
AnomalyWindows from Timeline & Anomaly Agent.

TOD band:

95% interval from TOD posterior.

Manual/External markers:

E.g. user‑flagged “interesting” times in UI; not necessary for hackathon, but design should allow it.

4.2 Merging Strategy
We create initial candidate intervals:

Start with all AnomalyWindows.

Intersect them with TOD band:

If withinTodBand is required for a candidate, we keep only overlapping portions (configurable).

Alternatively, we keep full anomaly windows but compute overlap fraction for scoring.

Merge overlapping / closely adjacent intervals:

If two windows are within a small gap (say, < 10 minutes), combine into a larger candidate:

start = min(starts), end = max(ends).

Optionally add a pure TOD candidate:

If there are no anomalies in the TOD band, we still produce at least one low‑confidence candidate centered on TOD to remind investigators of the uncertainty.

This is analogous to event correlation engines creating incident “groups” from multiple alerts and temporal proximity.

Result: a list of CandidateIntervals to score.

ts
CandidateInterval {
  id: string;
  caseId: string;
  start: string;
  end: string;
  anomalyWindowIds: string[];  // references
}
5. Feature Extraction per Candidate Interval
For each CandidateInterval, we build a feature vector describing how “interesting” this interval is across modalities.

5.1 Temporal Features
Duration (seconds, minutes).

Overlap with TOD band:

overlapFraction = |interval ∩ TOD_95| / |interval|.

Position relative to TOD:

e.g. centers before/after TOD median (for context).

5.2 Anomaly Features
From contributing AnomalyWindows:

Max/mean fused anomaly score in window.

Number of anomalies of different severities (ROUTINE, INTERESTING, CRITICAL).

Diversity of sources (phone, IoT, CCTV) in anomalies.

5.3 Event Features
From KeyEvents intersecting the interval:

Count of key events by type and severity.

Presence of specific patterns:

CCTV: person entering or leaving, victim with/without others.

Phone: calls/SMS to new contacts or blocked numbers.

IoT: unusual door patterns, alarms.

5.4 Forensic Context Features
From autopsy & TOD/hypotheses:

Whether there are injuries suggesting specific time‑critical mechanisms (e.g. asphyxia suggests events close to TOD).

Hypothesis baseline probabilities from Hypothesis Manager.

Whether interval contains events likely related to specific hypotheses:

Example: “high‑risk location” correlated with homicide.

These features align with general multi‑modal fusion frameworks that integrate time series with discrete events and clinical context to improve downstream tasks.

6. Scoring & Ranking – Multi‑Modal Fusion
We now convert CandidateInterval feature vectors into scores.

6.1 Global-Local Scoring Concept
We can think of this as a global-local hotspot detection problem:

Local: how strong are the anomaly and event signals in this specific interval?

Global: how unusual is this interval compared to the rest of the case, and how strongly does it connect to case hypotheses/TOD?

This is similar to Bayesian global-local hotspot models in high‑dimensional data (e.g. horseshoe priors shrinking noise globally while keeping strong local signals), and Bayesian hotspot identification / multiple testing frameworks that control for false discoveries.

7. Hypothesis Linkage & Causal Graph Integration
Once you have a score for each candidate interval, you:

Rank them by score.

For top K (e.g., 3–7) hotspots:

Create Causal Graph nodes:

Node type: EVIDENCE.

Label: “Digital Hotspot #1 (01:40–04:00)”.

Connect them to hypothesis nodes:

Edges with relation “SUPPORTS” / “CONTRADICTS” and strength from HypothesisImpact.

To compute HypothesisImpact:

Use an LLM (like DeepSeek‑V4) with a prompt summarizing:

The hotspot events (phone + CCTV + anomalies).

TOD band and autopsy conclusions.

Current hypothesis priors.

Ask it to reason:

How does this window affect probability of NATURAL vs ACCIDENT vs SUICIDE vs HOMICIDE?

Provide a reasoning chain and directional impact.

This is conceptually similar to multi‑modal fusion models that integrate time series with images/notes to improve classification tasks, but here the downstream task is “hypothesis support”.

8. XAI Layer: Explanation Narratives
After scoring, we want crisp textual explanations for investigators and the UI:

Components:

Time & TOD context:

“This hotspot occurs near the estimated time of death and overlaps 80% of the 95% TOD window.”

Anomaly summary:

“It includes a cluster of digital anomalies: unusual movement to a remote location, repeated calls to a new contact, and an unprecedented 90‑minute period of complete phone silence.”

Event correlation:

“CCTV shows the victim entering the building with another individual at 01:47, with no recorded exit prior to discovery.”

Hypothesis relationship:

“These patterns are more consistent with staged or coerced behavior than with the victim’s normal routine, supporting the homicide hypothesis.”

Many anomaly/correlation frameworks explicitly recommend providing summary narratives and root‑cause chains, not just scores.

We generate these via LLM:

Input: structured summary (scores, factors, events, hypothesis impacts).

Output: one or two paragraphs of explanation that the UI displays.

10. Implementation & Evolution
Implementation stack:

Python for fusion logic and scoring; models (if you learn weights) in scikit‑learn or light GBM.

LLM (DeepSeek‑V4, etc.) for HypothesisImpact reasoning & explanations.

Offline training:

If you ever collect retrospective labels (“this was the key interval”), you can:

Train logistic/GBM models on CandidateInterval features to learn weights.

Apply ideas from multiple testing / Bayesian hotspot identification to calibrate thresholds and FDR.

Alignment with research:

Follows forensic multi‑model fusion principles.

Follows multi‑modal time series fusion practices (intermediate / output fusion).

Inspired by global-local & Bayesian hotspot methods for robust selection.

Aligned with event correlation best practices for incident detection.
