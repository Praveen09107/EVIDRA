PART II — CATEGORY A: AUTOPSY & MEDICAL DOCUMENTS
A.1 What It Is
The official post-mortem examination report produced by a forensic pathologist. It is the single most important evidence category. It contains findings about cause of death, manner of death, time of death indicators (postmortem signs), injuries, toxicology, and body condition.

A.2 Accepted Formats
text
FORMAT           │ MIME TYPE                   │ INGESTION PATH     │ OCR REQUIRED
─────────────────┼─────────────────────────────┼────────────────────┼──────────────
PDF (text-based) │ application/pdf             │ Direct text extract│ NO
PDF (scanned)    │ application/pdf             │ PDF → image → OCR  │ YES
DOCX             │ application/vnd.openxml…    │ python-docx extract│ NO
JPG/PNG (scan)   │ image/jpeg, image/png       │ Direct OCR         │ YES
TIFF             │ image/tiff                  │ Direct OCR         │ YES
TXT              │ text/plain                  │ Direct read        │ NO
Real-world note: In the Indian forensic context, most autopsy reports arrive as scanned PDFs (physical reports photographed or flatbed-scanned). The OCR agent handles these with Tesseract (base) + LLM correction pass.

A.4 How It Flows Through the System
text
AUTOPSY DOCUMENT JOURNEY
═══════════════════════════════════════════════════════════════════════════════

1. UPLOAD (Layer 2)
   └── Investigator uploads PDF via Layer 4 UI
   └── Layer 2 receives file, computes SHA-256 hash
   └── Encrypts with AES-256-GCM, stores in MinIO bucket "aiventra-evidence"
   └── Inserts evidence_files record with:
         {file_type: "AUTOPSY_REPORT", mime_type: "application/pdf",
          file_size_bytes: N, sha256_hash: H, encryption_key_id: K}
   └── Custody event logged: "UPLOADED" → "SYSTEM:upload"

2. EVIDENCE PARSER (Agent 1)
   └── Receives: {file_id, s3_key, mime_type: "application/pdf"}
   └── Downloads encrypted file from MinIO, decrypts in-memory
   └── Detects: is this PDF text-based or image-based?
       ├── Text-based → extract text via pdfplumber
       │     └── Returns {raw_text, page_count, confidence: 0.95}
       └── Image-based → extract page images → route to OCR agent

3. OCR AGENT (Agent 2) [only if scanned]
   └── Receives: {images: [base64...], source: "AUTOPSY_REPORT"}
   └── Tesseract pass: extracts raw text per page
   └── Confidence check per page:
       ├── If avg confidence < 0.75 → LLM correction pass
       │     Prompt: "You are a forensic document OCR corrector.
       │     Fix OCR errors in this medical text: [text]
       │     Common fixes: 'rigor rnortis' → 'rigor mortis', etc."
       └── Returns {corrected_text, ocr_confidence_avg}

4. FORMAT NORMALIZER (Agent 3)
   └── Receives: {raw_text, source_type: "AUTOPSY_REPORT"}
   └── Detects autopsy report schema:
       SCHEMA A: AIIMS/NIMHANS format
       SCHEMA B: State hospital format
       SCHEMA C: Free-form narrative
       SCHEMA D: Fill-in-the-blank template form
   └── Maps to AIVENTRA canonical autopsy schema (all fields above)
   └── Returns: {canonical_autopsy_json, schema_detected, mapping_confidence}

5. AUTOPSY AGENT (Agent 4)
   └── Receives: {canonical_autopsy_json}
   └── LLM extraction pass (for any fields still needing natural language parsing):
       Prompt includes all 40+ fields listed above
   └── Runs pathology cross-checks:
       - COD vs injuries consistency
       - Toxicology vs manner consistency
       - Postmortem signs internal consistency
   └── Returns: AutopsyResult {
         mannerOfDeath, mannerConfidence,
         causeChain, toxicologyFindings,
         postmortemSigns, injuries, narrative
       }

6. TOD AGENT (Agent 5)
   └── Receives: AutopsyResult (specifically postmortemSigns, bodyParams, environmentFactors)
   └── Extracts: rectal_temp, ambient_temp, weight, rigor_stage, livor_stage, etc.
   └── Runs Henssge numeric solver + heuristic signs model + ML surrogate
   └── Monte Carlo uncertainty propagation → TOD posterior distribution
   └── Returns: TodResult {todPointEstimate, todWindow95, consistency, warnings}

7. HYPOTHESIS MANAGER (Agent 9)
   └── Receives AutopsyResult signals:
       {mannerOfDeath, defensiveWounds, toxicologyFindings, injuryList}
   └── Applies Bayesian likelihood ratios:
       - mannerOfDeath = HOMICIDE → LR_HOMICIDE = 15.0
       - defensiveWounds = true  → LR_HOMICIDE *= 3.2
       - drugsDetected   = trace → LR_SUICIDE += small positive weight
   └── Updates posterior probabilities
   └── Logs hypothesis snapshot to DB for trajectory chart

8. REASONING REPLAY (Agent 10)
   └── All 5 agents that processed this document log replay steps
   └── Each step: {trigger_type, input_summary, interpretation, confidence}
   └── Chain linked cryptographically

9. REPORT GENERATOR (Agent 11)
   └── Sections 2, 3 of final report populated from autopsy findings
   └── LLM-generated executive summary includes:
       "Postmortem examination indicates..."
PART III — CATEGORY B: CDR (CALL DETAIL RECORDS)
B.1 What It Is
The raw call and SMS log exported by a telecom operator for a specific phone number and time window. It records every call made/received, every SMS sent/received, every data session, with timestamps, duration, and cell tower IDs. This is the richest digital evidence source.

B.2 Accepted Formats
text
FORMAT                  │ MIME TYPE                        │ INGESTION PATH
────────────────────────┼──────────────────────────────────┼─────────────────────────
CSV (most common)       │ text/csv                         │ pandas read_csv
XLSX / XLS              │ application/vnd.openxml…         │ pandas read_excel
PDF (court-formatted)   │ application/pdf                  │ OCR + table detection
TXT (pipe/tab delimited)│ text/plain                       │ pandas read_csv (sep auto)
JSON (API export)       │ application/json                 │ json.loads
XML                     │ application/xml                  │ xml.etree
Most common real-world delivery: CSV or XLSX from Jio/Airtel/BSNL/Vi operators, forwarded by court order. Format varies per operator.

B.3 Raw Input Structure (CSV — Most Common)
Different telecom operators use different column names. The Format Normalizer maps all variants to the canonical schema. Here are the three most common raw formats:

B.4 Canonical CDR Schema (After Normalization)
The Format Normalizer maps all operator formats to this canonical structure:

python
CanonicalCdrEvent = {
    "event_id":            UUID,             # generated
    "case_id":             UUID,
    "source_msisdn":       str,              # MASKED: "98XXXXXX10"
    "event_timestamp":     datetime,         # ISO UTC — normalized from all date/time variants
    "event_type":          enum {            # MOC = mobile originating call
                             "MOC",          # MTC = mobile terminating call
                             "MTC",          # SMS_MO = SMS sent
                             "SMS_MO",       # SMS_MT = SMS received
                             "SMS_MT",       # DATA = data session
                             "DATA",
                             "MISSED",
                             "ROAMING_START",
                             "ROAMING_END"
                           },
    "duration_seconds":    int,              # 0 for SMS
    "counterparty_msisdn": str,              # MASKED
    "cell_tower_id":       str,              # raw tower ID
    "tower_latitude":      float,            # decimal degrees (if available)
    "tower_longitude":     float,
    "tower_name":          str,
    "imei":                str,              # MASKED: "35299950XXXXXXX"
    "data_volume_kb":      int | None,       # only for DATA events
    "raw_source_operator": str,              # "AIRTEL" | "JIO" | "BSNL" | "VI" | "UNKNOWN"
    "raw_row_index":       int,              # original row number for traceback
    "normalization_confidence": float        # 0.0–1.0
}
B.5 How CDR Is Used
text
CDR JOURNEY THROUGH AGENTS
═══════════════════════════════════════════════════════════════════════════════

1. UPLOAD (Layer 2)
   └── CSV/XLSX uploaded, SHA-256 computed
   └── Stored encrypted in MinIO
   └── evidence_files.file_type = "CDR"

2. EVIDENCE PARSER (Agent 1)
   └── Detects schema: CSV/XLSX → pandas load
   └── If PDF → routes to OCR agent for table extraction
   └── Extracts: row count, date range, MSISDN count
   └── Returns: {raw_dataframe_json (up to 50k rows), row_count, date_range}

3. FORMAT NORMALIZER (Agent 3)
   └── Operator detection:
       ├── Column headers match → AIRTEL / JIO / BSNL / VI
       └── Unknown → fuzzy column mapping + confidence score
   └── Date/time parsing: 30+ date format variants handled
       {"07/05/2026", "07-MAY-2026", "2026-05-07", "20260507", ...}
   └── Event type mapping:
       {"MOC" → "MOC", "O" → "MOC", "VOICE_OUT" → "MOC", "MO" → "MOC"}
   └── MSISDN normalization:
       {"09876543210", "9876543210", "+919876543210"} → canonical "919876543210"
   └── Tower coordinate enrichment:
       If tower_id present but lat/lon missing → look up in tower_registry DB table
   └── Outputs: [CanonicalCdrEvent] × N rows

4. DIGITAL TIMELINE AGENT (Agent 6)
   └── Receives: list of CanonicalCdrEvents
   └── Sorts by event_timestamp ASC
   └── Builds timeline structure:
       {
         events:          [TimelineEvent],
         time_range:      {start, end},
         unique_contacts: int,             # how many different numbers called
         silence_windows: [SilenceWindow], # gaps > threshold_minutes
         call_freq_by_hour: {0:N, 1:N, ...23:N}  # hourly activity histogram
       }
   └── Silence detection: gaps > 4h flagged as SILENCE_ANOMALY
   └── Pattern rules applied (15 rules):
       Rule 1:  Late-night calls (00:00–04:00)
       Rule 2:  Rapid sequential calls (< 3 min apart)
       Rule 3:  Very short calls (< 10 sec) — may be signaling
       Rule 4:  SMS burst (5+ SMS in < 30 min)
       Rule 5:  Last communication before silence
       Rule 6:  First communication after long silence
       Rule 7:  Calls to same number within TOD window
       Rule 8:  Roaming state change
       Rule 9:  IMEI change (device swap)
       Rule 10: New number first contact
       Rule 11: Call immediately after financial transaction
       Rule 12: Location jump without travel time
       Rule 13: Call during documented "victim last seen" window
       Rule 14: Calls to known emergency services
       Rule 15: Communication pattern break (normal daily pattern stops)

5. ANOMALY DETECTOR (Agent 7)
   └── Receives: TimelineOutput with flagged events
   └── ML models:
       - Autoencoder (reconstruction error): flags events that don't fit
         learned communication patterns
       - Isolation Forest: identifies statistically rare event combinations
   └── Per-event anomaly scores computed
   └── Threshold: anomaly_score > 0.75 → is_anomalous = true

6. HOTSPOT CORRELATOR (Agent 8)
   └── Receives: AnomalyOutput + TOD window
   └── Finds: which CDR anomalies fall inside TOD window?
   └── Creates hotspot windows where:
       - Multiple anomalous CDR events cluster temporally
       - AND overlap with TOD credible interval

7. HYPOTHESIS MANAGER (Agent 9)
   └── CDR-derived signals used:
       - last_contact_before_death: WHO was last contact?
       - communication_cessation_time: WHEN did phone go silent?
       - late_night_contact_pattern: unusual nighttime activity?
       - contact_frequency_anomaly: sudden change in calling pattern?
   └── These signals adjust Bayesian posteriors:
       - Last contact 30min before estimated TOD → LR_HOMICIDE boost
       - Complete communication cessation = TOD → consistent with all hypotheses
       - Unusual contact pattern change days before death → LR_HOMICIDE moderate boost

C.5 How Location Data Is Used
text
LOCATION DATA USAGE
═══════════════════════════════════════════════════════════════════════════════

DIGITAL TIMELINE AGENT:
  Key analyses performed on location data:
  ─────────────────────────────────────────
  1. PRESENCE AT SCENE
     └── Was the phone/device at the crime scene location
         during the TOD window?
     └── Algorithm: compute distance from each location fix
         to scene GPS coordinates. If distance < 200m
         during TOD window → flag as SCENE_PRESENCE

  2. MOVEMENT TRAJECTORY
     └── Build movement path: sequence of lat/lon points
     └── Detect: was the device moving or stationary?
     └── Speed between consecutive points:
         speed = distance(p1, p2) / time_delta
         If speed > 150 km/h → IMPOSSIBLE_MOVEMENT flag
         If speed implies teleportation → LOCATION_JUMP flag

  3. UNUSUAL LOCATION
     └── Build "home range" from 30-day history (if available)
     └── Locations outside normal range → UNUSUAL_LOCATION flag

  4. LAST KNOWN LOCATION
     └── Final GPS fix before communication silence
     └── Used as: "last confirmed alive at [coordinates]"
     └── Fed to TOD Agent as additional prior constraint

  5. ALIBI CORROBORATION / CONTRADICTION
     └── If suspect phone location contradicts stated alibi
         → flag for investigator review

HYPOTHESIS MANAGER:
  ─────────────────────────────────────────
  - device_at_scene_during_tod_window → TRUE  → LR_HOMICIDE boost
  - device_left_scene_immediately_after_estimated_tod → TRUE → LR_HOMICIDE strong boost
  - device_never_left_home → TRUE → LR_SUICIDE / NATURAL boost
  - impossible_movement detected → flag for manual review

D.1 What It Is
Transaction logs from bank accounts, UPI apps (PhonePe, GPay, Paytm), credit cards, or NEFT/RTGS records obtained under court order. Shows financial behavior patterns immediately before death.

D.3 Raw Input Structure

E.5 Sensitive Content Handling
text
DEVICE DATA SPECIAL RULES
═══════════════════════════════════════════════════════════════════════════════

1. CONTENT MASKING
   All actual message text is:
   - PII-stripped (names replaced with [CONTACT_1], [CONTACT_2], etc.)
   - Stored in masked form only in canonical events
   - Original text retained ONLY in encrypted raw storage (MinIO)
   - Agents receive: content_masked, content_sentiment, NOT original text

2. SENTIMENT ANALYSIS
   Each message passes through sentiment classifier:
   - Gemini Flash with prompt:
     "Classify this message as: POSITIVE, NEUTRAL, NEGATIVE,
     DISTRESSED (signs of hopelessness/farewell), or
     THREATENING (violence/coercion). Output JSON only."
   - Result: {sentiment, confidence, flags: [str]}
   - DISTRESSED flag → Hypothesis: suicide signal
   - THREATENING flag → Hypothesis: homicide signal

3. DELETED MESSAGE DETECTION
   If source is Cellebrite UFED or SQLite dump:
   - Can detect messages marked as deleted but not overwritten
   - message_deleted = true → STRONG anomaly flag
   - Deleted messages near TOD → very high anomaly score

4. SEARCH QUERY SENSITIVITY
   If SEARCH_QUERY events present:
   - Queries are redacted in all reports/UI except to LEAD_INVESTIGATOR+
   - Content pattern matching: "how to" + violence keywords → flag
   - Stored in PII vault, access logged

PART X — DATA VALIDATION & QUALITY GATES
Every piece of evidence passes through three quality gates before being used:

text
GATE 1: INTEGRITY (Layer 2)
─────────────────────────────────────────────────────────────────────────────
Check 1: SHA-256 hash verification (file not corrupted in transit)
Check 2: File size > 0 bytes
Check 3: MIME type matches file extension (no spoofing)
Check 4: Malware scan (ClamAV) — quarantine if infected
Check 5: File size < 500MB per file (system limit)
Result: PASS → proceed │ FAIL → QUARANTINED, never processed

GATE 2: PARSABILITY (Evidence Parser + Normalizer)
─────────────────────────────────────────────────────────────────────────────
Check 1: File readable (not password-protected, not corrupted)
Check 2: Encoding detected and decodable (UTF-8, UTF-16, CP-1252, ISO-8859)
Check 3: At least 1 data row parseable (not completely empty)
Check 4: Date fields parseable (at least 80% of rows have valid timestamps)
Check 5: Schema confidence > 0.50 (Normalizer can map to canonical schema)
Result: PASS → proceed │ PARTIAL → proceed with warnings │ FAIL → skip this file

GATE 3: USEFULNESS (Per-Agent)
─────────────────────────────────────────────────────────────────────────────
For TOD Agent:
  Check: At least one of {rectal_temp, rigor_stage, livor_stage} present
  If NONE → mode = "SIGNS_CONSTRAINED", warning added

For Digital Timeline Agent:
  Check: At least 5 events with valid timestamps
  If FAIL → timeline output = "INSUFFICIENT_DATA", agent marked SKIPPED

For Hypothesis Manager:
  Check: At least 2 Bayesian signals available (from any category)
  If FAIL → posteriors = uniform {0.2 each}, HIGH severity warning added
