1. Timeline & Anomaly Agent – Purpose and Role
1.1 Purpose
This agent is the “digital nervous system” of the case. It ingests heterogeneous digital signals (phone logs, app usage, network logs, IoT sensors, CCTV metadata, etc.), converts them into a unified time series, and then detects unusual patterns and time windows that might be relevant to the death.

Its responsibilities:

Normalize multi‑source digital evidence into a case timeline with per‑time‑bucket features.

Learn normal behavior for that person/context (within‑case or across reference datasets).

Detect anomalies using a hybrid of:

Deep autoencoder for multivariate time series (reconstruction error).

Isolation Forest (IF) for statistical outlier scores.

Fuse these signals into anomaly windows with scores and explanations that can be visualized on the timeline and passed to the Hotspot/Correlation Agent.

This mirrors modern network/IoT anomaly frameworks that combine autoencoders + Isolation Forest for more robust anomaly detection than either alone.

4. Modules in Detail
4.1 Event Normalization & Enrichment
Goal: bring heterogeneous events into a canonical schema and enrich them with useful derived attributes.

Steps:

Normalization: map raw source fields to canonical fields (duration, location, device, etc.).

Context enrichment:

Geocoding for “home/work/other” classification from GPS.

Flagging night vs day, weekend vs weekday.

Mapping app IDs to categories (social, financial, entertainment).

Device/session grouping:

Merge low‑level events into sessions (e.g. app session, screen‑on session) for better semantic features.

Most of this is deterministic backend logic and runs before ML; the autoencoder sees a relatively clean numerical feature tensor.

4.3.2 Training Strategy
Training mode options:

Person‑specific unsupervised:

Population‑level:

Optimization:

Minimize reconstruction loss (mean squared error or robust losses).

Apply regularization and compression strategies (pruning, quantization) as in efficient AE research to keep models fast for real‑time scoring.

4.4 Isolation Forest Module
4.4.1 Rationale
Isolation Forest (IF) is a classical, unsupervised anomaly detector that isolates anomalies by random partitioning and uses path lengths to compute anomaly scores.

It is:

Efficient for high‑dimensional data.

Complementary to AE: AEs model continuous manifolds, IF detects statistical outliers that may be rare but not poorly reconstructed.

Recent security and IoT work shows hybrid AE + IF frameworks outperform either component alone for detecting various anomalies (DoS, ransomware, IoT traffic drift).

4.4.2 Training
We train an IF model on a “normal” subset of timeline buckets:

Use pre‑case baseline or population normal dataset.

Parameters:

Number of trees (e.g. 100–300).

Subsample size per tree.

Contamination rate (expected fraction of anomalies).

5. Outputs – Anomaly Windows & Explanations
5.1 AnomalyWindow Schema
We already sketched something like this; here is a detailed version:

ts
AnomalyWindow {
  id: string;
  caseId: string;
  start: string;
  end: string;
  aeScore: number;          // aggregated AE score (e.g., max or mean over window)
  ifScore: number;          // aggregated IF score
  fusedScore: number;
  factors: AnomalyFactor[]; // top-k feature factors
  sourceBreakdown: SourceContribution[];
  label?: "ROUTINE" | "INTERESTING" | "CRITICAL";  // severity
  explanation: string;      // narrative, for UI
}
ts
AnomalyFactor {
  feature: string;          // "callCount", "distanceMovedMeters", ...
  contribution: number;     // relative importance within window
  direction: "HIGHER_THAN_USUAL" | "LOWER_THAN_USUAL";
}

SourceContribution {
  source: "PHONE" | "APP" | "IOT" | "CCTV" | "NETWORK";
  weight: number;           // fraction of fused score attributable to this source
}
Feature‑level importance can be derived by aggregating reconstruction error per feature from the AE, plus any feature importance techniques applied on an interpretable variant of IF, similar to ExIFFI or importance analyses in IF frameworks.

5.2 XAI Narratives
A small LLM pass can turn the numeric explanation into natural language:

“Between 02:10 and 02:45, the victim’s phone showed an unusually high number of short calls to unknown numbers and a complete absence of typical late‑night app usage, which differs from their baseline pattern. Location data also suggests a longer‑than‑usual stationary period at an unfamiliar address.”

This is conceptually similar to explanation‑oriented anomaly detection pipelines in security, where feature importance is used to justify flagged anomalies.

6. Modes: Training / Scoring / Streaming
6.1 Offline Training Mode
Train or fine‑tune AE and IF models on:

Pre‑case normal segments (if available).

Population‑level normal datasets by source (phone, IoT, etc.).

Hyperparameter tuning using:

Reconstruction error distributions and F1/ROC on labeled anomalies (if any).

Benchmarks similar to those in time series anomaly detection surveys (point & collective anomalies).

6.2 Batch Scoring Mode
Used for initial case ingest:

Build full timeline for case (from earliest to latest events).

Compute AE & IF scores for all buckets.

Extract anomaly windows and persist them.

Send them to Hotspot Agent and UI.

6.3 Streaming/Online Mode
For live monitoring or incremental updates:

As new digital events arrive:

Update relevant buckets.

Compute incremental features.

Run AE & IF scoring only on affected windows.

Update anomaly windows (extend, split, merge) accordingly.

Modern time‑series anomaly detection focuses heavily on online detection; the same architectural ideas (incremental scoring, windowing, concept drift handling) apply here.

7. Interaction with Other Agents
7.1 TOD Agent
The TOD Agent’s posterior window defines a focus band on the timeline.

The Timeline & Anomaly Agent can:

Mark anomalies falling within the TOD 95% interval more prominently (higher effective priority).

Provide to TOD Agent summary stats about anomaly density near candidate TODs for feedback loops if you want.

7.2 Hotspot & Correlation Agent
This agent takes:

AnomalyWindows (with fusedScore).

TOD posterior window.

Other signals (CCTV, phone, IoT correlation events).

And produces Hotspots, which include the anomaly windows as one of the contributing components. The anomaly windows effectively provide candidate suspicious intervals to be intersected with TOD and other evidence.

7.3 Causal Graph & Hypothesis Manager
Anomalies can be turned into candidate evidence nodes, e.g. “Unusual phone silence between 22:00–23:00” or “Unexpected movement to unfamiliar location at 03:13”.

Each anomaly window can map to a node with edges to hypotheses (suicide vs homicide vs accident) via LLM reasoning.

8. Example Flow – From Raw Events to AnomalyWindow
8.1 Scenario
Victim usually:

Has messaging & social app activity until 23:30.

Minimal activity between 00:00–06:00.

On case night:

From 01:40–02:10, there are 12 short outgoing calls to a new contact and 6 location pings moving to a remote industrial area.

From 02:10–04:00, complete silence from phone and apps, unlike baseline where there is usually at least some late‑night chat usage.

IoT logs show door open/close at 01:42 and not again until morning.

8.2 Processing
Events → buckets: 5‑minute buckets from 00:00 to 08:00.

Feature vectors show:

Spike in callCount and distanceMoved between 01:40–02:10.

Near‑zero activity but unusual pattern of “hard silence” between 02:10–04:00 compared to baseline.

AE scoring:

Normal windows reconstruct well (low error).

01:40–02:10 windows show high error due to unusual combined call + movement pattern.

02:10–04:00 windows show high error because AE expects occasional messaging or app background activity from baseline but gets zero.

IF scoring:

Points for 01:40–02:10 and 02:10–04:00 lie in sparse regions of the feature space; IF marks them with high anomaly scores.

Fusion & windows:

Fused scores exceed threshold from ~01:40–04:00, with two sub‑peaks:

Activity spike.

Silence region.

The agent merges them into one extended AnomalyWindow or two adjacent ones depending on configuration.

9. Implementation & Operational Considerations
Tech stack:

Python: PyTorch/TF for AE, scikit‑learn for IF.

Optionally accelerate with ONNX/TensorRT for real‑time scoring.

Model management:

Version AEs & IFs (timeline_ae_vX, timeline_if_vY).

Store version IDs in each AnomalyWindow for auditability.

Evaluation:

Use benchmark datasets and detection metrics (Precision/Recall/F1, ROC AUC) as in network/IoT anomaly detection studies to validate the hybrid approach.
