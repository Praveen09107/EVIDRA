AIVENTRA — Complete End-to-End Pipeline: A to Z
CHAPTER 1: SYSTEM OVERVIEW
1.1 What AIVENTRA Is
AIVENTRA is a multi-agent AI forensic intelligence platform that ingests raw case files (autopsy reports, call detail records, financial records, device data) and autonomously produces a structured forensic analysis — including time of death estimation, communication anomalies, financial motive scoring, unified timeline, ranked hypotheses, and a fully auditable reasoning replay report.

1.2 Design Philosophy
No black box. Every inference is traced to a specific evidence item and reasoning step.

Physics first, ML second. Classical forensic models (Henssge, PMI heuristics) anchor ML outputs.

Uncertainty over false precision. All estimates carry explicit confidence intervals.

Explainability by design. SHAP values, narrative generation, and contradiction detection are first-class outputs.

# Gateway checks:

def validate_case_files(case_id: UUID) -> ValidationResult:
    files     = db.fetch_files(case_id)
    doc_types = {f.doc_type for f in files}

    warnings = []
    if "AUTOPSY_REPORT" not in doc_types:
        warnings.append("No autopsy report — TOD and manner analysis will be limited")
    if "CDR" not in doc_types:
        warnings.append("No CDR data — communication analysis will be skipped")
    if not files:
        return ValidationResult(valid=False, error="No files uploaded")

    return ValidationResult(valid=True, warnings=warnings)

CHAPTER 7: OCR WORKER AGENT
7.1 Two-Stage OCR Strategy
text
Stage 1: Tesseract (local, free)
  → Used for: clear printed text, typed reports
  → Languages: eng + hin (configurable)
  → Config: --oem 3 --psm 6 (assume uniform block)
  → Threshold: confidence > 0.75

Stage 2: Google Vision API (cloud fallback)
  → Used when: Tesseract confidence < 0.75
  → Use cases: handwritten notes, faded ink, photographs
  → Returns: full text annotations with bounding boxes

Stage 3: Hybrid merge
  → Page-level decision: if Vision > Tesseract confidence → use Vision
  → Stitch pages into single text document
  → Store: s3_key = cases/{case_id}/ocr/{file_id}/extracted.txt
7.2 OCR Quality Scoring
python
def score_ocr_quality(text: str, page_count: int) -> float:
    words          = text.split()
    if not words: return 0.0

    # Heuristics for OCR quality:
    real_word_ratio = sum(1 for w in words if w.isalpha()) / len(words)
    avg_word_len    = np.mean([len(w) for w in words])
    chars_per_page  = len(text) / max(page_count, 1)

    score  = 0.0
    score += real_word_ratio * 0.5           # most words should be real
    score += min(1.0, avg_word_len / 6) * 0.3 # avg word 5-7 chars is good
    score += min(1.0, chars_per_page / 500) * 0.2  # reasonable density

    return round(score, 3)

text

CHAPTER 10: CDR ANALYZER AGENT (M-06)
10.1 Full Analysis Pipeline
text
Input: canonical_cdr_events for case_id (from DB)

STEP 1 — Behavioral Baseline:
  → Compute 30-day baseline per MSISDN:
      daily_call_mean, daily_call_std
      active_hours, typical_silent_hours
      top_contacts, tower_entropy
      hourly_distribution

STEP 2 — Last Activity Detection:
  → Find last CDR event for victim's MSISDN
  → Record: last_call_timestamp, last_call_counterparty, last_tower

STEP 3 — Silence Window Detection:
  → From last CDR event to discovery time = silence duration
  → Z-score vs baseline max-silence
  → If overlap with TOD window → severity bump
  → Output: { duration_h, z_score, anomaly_score, severity }

STEP 4 — Contact Escalation:
  → Per-contact call frequency in 72h before last event vs 30-day baseline
  → Flag contacts with escalation_ratio ≥ 2.5 and ≥ 3 calls
  → Output: [{ contact_msisdn, escalation_ratio, severity }]

STEP 5 — Tower Proximity Analysis:
  → For flagged contacts: find their last tower during TOD window
  → Haversine distance from victim's address
  → Flag if within 2× tower radius (~3km urban)

STEP 6 — IMEI Analysis:
  → Check for IMEI changes (SIM swap or new phone)
  → Flag if IMEI changes within 48h of death

STEP 7 — Network Graph:
  → Build contact graph for victim
  → Identify bridge nodes (contacts connecting victim to unknown parties)

Output: CdrAnalysisResult
{
  last_activity:         { timestamp, counterparty, tower_id }
  silence_window:        { start, end, duration_h, anomaly_score, severity }
  escalation_flags:      [EscalationFlag]
  proximity_flags:       [ProximityFlag]
  imei_anomalies:        [ImeiAnomaly]
  tower_sequence:        [TowerEvent]
  analysis_flags:        [string]  (high-level flags for UI)
  confidence:            float
}
CHAPTER 11: FINANCIAL ANALYZER AGENT (M-07)
11.1 Full Analysis Pipeline
text
Input: canonical_financial_events for case_id (from DB)

STEP 1 — Baseline Construction:
  → 90-day rolling stats per account:
      monthly_mean_amount, monthly_std_amount
      known_counterparties, typical_txn_categories
      max_single_txn_historical
      cash_vs_digital_ratio

STEP 2 — Point Anomaly Detection (Isolation Forest):
  → Feature vector per transaction:
      amount, amount_zscore, log_amount
      hour, is_weekend, is_cash
      is_round_number, balance_ratio
  → IF score → anomaly_probability [0,1]
  → Flag if anomaly_probability > 0.65

STEP 3 — Sequence Anomaly Detection (BiLSTM AE):
  → 14-day sliding window of daily txn features
  → Reconstruction error > 0.038 → sequence anomaly
  → Identifies pattern shifts (e.g. sudden cash escalation week)

STEP 4 — Motive Pattern Scoring:
  → Checks within 72h before TOD for:
      LARGE_CASH_WITHDRAWAL (>3× monthly mean)
      REPEATED_CASH_ESCALATION (multiple cash txns escalating)
      LARGE_TRANSFER_TO_UNKNOWN (new counterparty, large amount)
      INSURANCE_POLICY_ACTIVITY ("insurance", "policy" in narration)
      DEBT_REPAYMENT_SPIKE (EMI/loan >1.5× normal)
  → Computes motive_score [0,1]

STEP 5 — LLM Narrative:
  → Sends anomalies + motive flags to GPT-4o
  → Gets plain-English financial analysis narrative
  → Used directly in Reasoning Replay report

Output: FinancialAnalysisResult
{
  baseline_stats:       BaselineStats
  point_anomalies:      [PointAnomaly]
  sequence_anomalies:   [SequenceAnomaly]
  motive_flags:         [MotiveFlag]
  motive_score:         float
  llm_narrative:        string
  total_anomalous_amount: float
  confidence:           float
}
CHAPTER 12: TOD AGENT (M-08)
12.1 Full Estimation Pipeline
text
Input: AutopsyExtractionMinimal + SceneData + TemperatureMeasurements

STEP 1 — Mode Selection:
  → If rectal_temp available → HYBRID mode
  → If only signs available → SIGNS_CONSTRAINED mode
  → If only prior/timeline → PHYSICS_ONLY with wide prior

STEP 2 — Physics Core (Henssge Numeric Solver):
  → Inputs: temp_rectal, temp_ambient, weight_kg, correction_factor
  → Iteratively solves cooling equation via scipy.brentq
  → Output: PMI_mean_hours, CI_lower_95, CI_upper_95
  → Tolerance: ±2.8h (PMI<10h), ±4.8h (10-20h), ±7.4h (>20h)

STEP 3 — Heuristic Sign Likelihoods:
  → Maps rigor/livor/decomp stages to truncated Gaussian distributions
  → L_rigor(t), L_livor(t), L_decomp(t) over PMI grid

STEP 4 — ML Surrogate (RF/GBM Ensemble):
  → 13-feature vector → RF (500 trees) + GBM (300 trees)
  → Ensemble weights: 0.55 RF, 0.45 GBM
  → Output: PMI_ml_mean, PMI_ml_std (from RF tree variance)

STEP 5 — Prior Construction:
  → If last_seen_alive + found_dead known:
        Uniform prior over that interval + soft tails
  → Else: Broad uniform 0–72h before discovery

STEP 6 — Bayesian Monte Carlo Fusion (20,000 samples):
  → For each PMI candidate t:
        log_weight = log_prior(t)
                   + log_L_physics(t)
                   + 0.7 * log_L_ml(t)
                   + log_L_rigor(t)
                   + log_L_livor(t)
                   + log_L_decomp(t)
  → Normalize weights → posterior distribution
  → Extract: median, 95% CI, 68% CI

STEP 7 — Consistency Check:
  → Compare posterior band with each sign's plausible PMI range
  → CONSISTENT / INCONSISTENT / INSUFFICIENT_DATA per sign

STEP 8 — SHAP Explainability:
  → TreeSHAP on RF model for feature contributions
  → Map to TodFeatureContribution[] with direction + description

STEP 9 — LLM Narrative:
  → Numeric summary → GPT-4o → plain English TOD explanation

Output: TodResult
{
  mode:              HYBRID | PHYSICS_ONLY | SIGNS_CONSTRAINED
  todPointEstimate:  ISO timestamp
  todWindow95:       { start, end }
  todWindow68:       { start, end }
  componentContributions: [...]
  consistency:       { rigor, livor, algor, decomposition }
  featureImportance: [TodFeatureContribution]
  warnings:          [string]
  explanationNarrative: [string]
}
CHAPTER 13: TIMELINE & ANOMALY AGENT (M-09)
13.1 Full Pipeline
text
Input: All Tier-2 agent outputs (CDR, Financial, Autopsy, TOD)

STEP 1 — Unified Timeline Construction:
  → Merges CDR events + financial events + TOD window into
     a single chronological event stream
  → Resolution: 15-minute bins
  → Feature matrix: (T, 8) → call_count, sms_count, data,
                               contacts, fin_txn, fin_amount,
                               tod_prob, silence_indicator

STEP 2 — LSTM Autoencoder Anomaly Detection:
  → Sliding window: 32 bins (8h) × stride 4
  → Reconstruction error per window vs threshold 0.045
  → Flag windows: error > threshold → anomaly window
  → Severity: HIGH (>2×threshold), MEDIUM (1.4×), LOW (1×)

STEP 3 — Isolation Forest Cross-Modal:
  → Per-bin feature vector across modalities
  → IF score → point anomalies at specific timestamps
  → Cross-references with CDR silence windows

STEP 4 — TOD Intersection:
  → Find all anomaly windows that overlap TOD 95% credible interval
  → These are "critical windows" — high investigative priority

STEP 5 — Contact Co-location:
  → For each flagged contact from CDR Agent:
        Check if their tower places them in victim's cell cluster
        during any anomaly window
  → Co-location flag: contact + window + distance

STEP 6 — Anomaly Window Ranking:
  → Score each window:
        base_score = reconstruction_error / threshold
        + 0.30 if overlap with TOD window
        + 0.20 if flagged contact co-located
        + 0.15 if financial anomaly in same window
  → Sort by combined score

Output: TimelineAnomalyResult
{
  unified_timeline_bins:  int
  anomaly_windows:        [AnomalyWindow]
  critical_windows:       [AnomalyWindow]  (TOD overlap)
  co_location_flags:      [CoLocationFlag]
  tod_intersection_count: int
  confidence:             float
}
CHAPTER 14: HOTSPOT ENGINE (M-10)
14.1 Full Pipeline
text
Input: TOD result + Timeline result + CDR result + Financial result

STEP 1 — Temporal Hotspot:
  → Intersect:
        TOD 95% window
        CDR silence window
        Financial anomaly window (72h before TOD)
        Critical anomaly windows from Timeline Agent
  → Hotspot_time_window = broadest consistent overlap

STEP 2 — Contact Hotspot:
  → From CDR: escalation_flags + proximity_flags
  → Score each contact:
        score += 0.35 * (escalation_ratio / 4)        # call escalation
        score += 0.30 * (1 if proximate else 0)        # physical proximity
        score += 0.20 * (1 if last_call_before_silence) # last known contact
        score += 0.15 * (financial_link_score)          # financial connection
  → Rank contacts by hotspot score

STEP 3 — Spatial Hotspot:
  → From tower data: identify cell towers with high activity during hotspot window
  → Map overlapping towers between victim and top contacts
  → Output: tower_ids, lat/lon cluster

STEP 4 — Cross-Modal Confidence:
  → Count how many independent evidence streams point to same hotspot:
        Each supporting stream adds +0.15 to hotspot confidence
  → Max confidence capped at 0.95

STEP 5 — Motive-Opportunity Matrix:
  → For each contact in hotspot:
        Motive:      financial_link_score + escalation_ratio > threshold
        Opportunity: proximate during TOD window
        Score:       motive × opportunity × normalization

Output: HotspotResult
{
  temporal_hotspot:     { start, end, supporting_streams }
  contact_hotspots:     [ContactHotspot ranked by score]
  spatial_hotspot:      { tower_ids, lat, lon, radius_km }
  motive_opportunity:   [{ contact, motive_score, opportunity_score }]
  confidence:           float
}
CHAPTER 15: HYPOTHESIS MANAGER (M-11)
15.1 Full Scoring Pipeline
text
Input: All prior agent outputs

STEP 1 — Evidence Flag Extraction:
  → From autopsy:   "blunt_force_trauma", "defensive_wounds",
                    "body_not_moved", "natural_disease_history", etc.
  → From CDR:       "phone_silence_coincides_tod", "contact_escalation"
  → From financial: "large_cash_withdrawal", "insurance_activity"
  → From hotspot:   "contact_proximity_tod"

STEP 2 — Bayesian Log-Additive Scoring:
  → Start: uniform prior log(0.2) for each of 5 hypotheses
  → For each evidence flag:
        log_score[hypothesis] += EVIDENCE_WEIGHT[flag][hypothesis]
  → Softmax-normalize → probability scores

STEP 3 — Contradiction Detection:
  → Check for logical impossibilities:
        HOMICIDE + hesitation_wounds          → contradiction
        NATURAL + blunt_force_trauma          → contradiction
        SUICIDE + body_not_moved + no_note    → potential contradiction
  → Each contradiction: { type, description, severity, affected_hypotheses }

STEP 4 — Supporting / Weakening Evidence Lists:
  → For primary hypothesis: collect all flags with positive weight
  → For primary hypothesis: collect all flags with negative weight

STEP 5 — LLM Recommendation Generation:
  → Sends scores + evidence + contradictions to GPT-4o
  → Gets: 5 ranked recommended_actions for investigators

STEP 6 — Confidence Adjustment:
  → Base: primary_score from Bayesian model
  → Penalize: -0.05 per HIGH contradiction
  → Penalize: -0.03 per missing key evidence (no toxicology, no device data)
  → Final: primary_confidence [0,1]

Output: HypothesisResult
{
  scores:                { HOMICIDE, SUICIDE, ACCIDENT, NATURAL, UNDETERMINED }
  primary_hypothesis:    string
  primary_confidence:    float
  supporting_evidence:   [string]
  weakening_evidence:    [string]
  contradictions:        [Contradiction]
  recommended_actions:   [string]
  confidence:            float
}
CHAPTER 16: REASONING REPLAY AGENT (M-12)
16.1 Full Pipeline
text
Input: ALL agent_results + ALL agent_tasks for pipeline_run_id

STEP 1 — Step Collection:
  → Queries replay_steps table for all steps in this pipeline_run
  → Orders by global_index
  → Groups by agent_id

STEP 2 — Global Step Indexing:
  → Assigns sequential global_index across all agents
     in tier execution order:
        evidence_parser steps → 1, 2, 3...
        ocr steps            → next...
        autopsy steps        → next...
        ...continuing through hypothesis_manager

STEP 3 — Reasoning Gap Detection:
  → A gap = a decision in one agent that cannot be traced to
     any output of its prerequisite agents
  → For each step: verify evidence_ids[] map to actual case_files
  → For each inference: check parent_step_ids[] are non-empty
  → Gaps: { gap_before_step, description, severity }

STEP 4 — Contradiction Detection:
  → Cross-agent consistency checks:
        TOD window vs CDR silence: consistent?
        Autopsy manner vs wound pattern: consistent?
        Financial motive window vs TOD: temporally consistent?
        Hypothesis score vs autopsy confidence: aligned?
  → Output: [Contradiction]

STEP 5 — Confidence Aggregation:
  → Per-agent: harmonic mean of all step confidences in that agent
  → Overall: harmonic mean of all step confidences
              − 0.05 per HIGH contradiction
              − 0.03 per reasoning gap
  → Overall: capped [0.10, 0.99]

STEP 6 — DAG Construction:
  → Nodes: each replay step
  → Edges: parent_step_ids links
  → Used by frontend DAG viewer

STEP 7 — LLM Narrative Generation:
  → Sends structured summary to GPT-4o:
        TOD estimate + window
        Manner of death + confidence
        CDR key findings
        Financial key findings
        Primary hypothesis + confidence
        All contradictions
        All gaps
  → Gets: 500-800 word plain English forensic narrative
  → Temperature: 0.2 (slight variation for readability)

STEP 8 — Step Index Table:
  → Assembles UI-ready table:
        step_id, global_index, agent_id, trigger_type,
        timestamp, input_summary, interpretation,
        confidence, evidence_ids, warnings, duration_ms

Output: ReasoningReplayReport

  pipeline_run_id:         UUID

  reasoning_gaps:          [Gap]

}

17.2 Retry / Failure Policy
text
Per agent task:
  Max attempts:       3
  Backoff:            10s → 30s → 90s
  Non-retryable:      INVALID_DATA, SCHEMA_ERROR
  After max retries:  task = FAILED, pipeline = PARTIAL

Pipeline PARTIAL rules:
  If hypothesis_manager or reasoning_replay failed → pipeline = FAILED
  If any Tier 2 agent failed but others succeeded → pipeline = PARTIAL
  Partial results still returned with clear PARTIAL flag in UI
